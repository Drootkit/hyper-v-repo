# Read Technical Articles

## Instructions

## 核心目标

高效提取技术文章的核心价值，严格遵循“总-分”结构输出。帮助用户以最小时间成本掌握全局结论，并按原文脉络深度还原功能实现与技术原理，确保技术细节可追溯、逻辑链条完整。

```
“段”优先级定义：
1. 文章一级标题（Section）
2. 二级标题（Subsection）
3. 若无标题，按自然段聚合
```

### 输出格式规范（总-分结构）

- 所有技术总结必须严格采用“总-分”格式组织内容，禁止打乱原文逻辑顺序或跳跃式归纳。
- 禁止在报告中，使用大量的emoji、恭维点评语气或非技术性修饰词，确保内容专业、严谨。
- 除非用户要求，否则不能擅自将分析内容保存到文件中。保存格式为 markdown，文件名称和分析的文章保持一致，前面加上"LLM分析 "。

0. 基础信息：

```
文章标题：
作者/来源：
发布时间（如提供）：
分析时间：
技术领域标签：
原文链接（如提供）：
```

1. 总体摘要（总）

- 总结文章每段核心结论：针对每段，用不超过 200 个汉字概括最核心的观点、技术突破或最终成果
- 关键要点提炼：提取核心结论，明确标注技术栈/平台依赖、创新点、实践价值或关键数据
- 概括文章研究思路：将文章研究思路进行抽象概括，用一段话总结文章的技术路线。

比如：

```
总体摘要：

  第1段：xxxxx
  第2段：xxxxx
  第3段：xxxxx
  第n段：xxxxx

  总结：该文章对xxxx，从xxx方面，展开了研究，使用了xxx，涉及了xxxx，...... 最后文章认为xxx/得出结论是xxxx。（你需要根据不同的文章类型，灵活使用）
  
  根据文章描述的内容，给出 mermaid 格式的流程图，分别
    - 展示文章的研究思路。
    - 展示文章中描述的内容（可以是多张图或者是0张图，依据是文章中提到的实际技术内容）。
  
  当文章包含下面的内容时，可以生图，其他情况你认为合适也可以出图：
    - 架构设计
    - 数据流
    - 系统流程
    - Pipeline
    - ......
```

2. 分段详解（分）：依照文章的章节、段落或技术推演顺序进行逐段拆解与重述。（遵循下面的要求和规范）

- 每一部分需保留原文的上下文依赖关系，确保技术演进路径清晰连贯。
- 将当前原文内容进行条理化输出，比如原文是一段长描述，你需要保持原文的技术细节和逻辑顺序，在避免过度概括或删减的情况下，按照一定的调理进行重述。必要的时候使用mermaid等流程图来增强说明。
  ```
  数据处理规则：
  - 必须提取：数据集、指标、基线模型、等其他可以使用表格展示的内容
  - 表格必须转为 markdown 表格
  ```
- 对文章中出现的代码段，将相关的代码进行注释说明，帮助理解代码的功能和作用；其他的代码使用省略号或者伪代码表示。
- 段落标题需与原文保持一致或进行精准语义映射，便于对照查阅。
- 文章中如果有一些专业术语或者技术细节没有进行详细说明，你需要在总结的过程中进行补充说明，帮助理解这些术语或细节的含义和作用（通过标注的方式，并且标明来源比如文章本身或者LLM新增的）。

### 深度内容撰写要求（原理与功能解析）

针对文章中涉及的功能实现、机制设计、架构原理或算法逻辑，必须进行详细总结，严禁概括性略写或一笔带过。具体表述需严格遵循以下话术范式：

```
本段讲述了 [功能/模块名称] 的实现机制。实现过程使用了 [具体技术/方法/工具] 方式，利用了 [底层框架/系统/协议] 的 [具体原理]，同时还需要满足 [前置条件1]、[前置条件2]、[前置条件3]。其中 [核心逻辑/关键步骤] 的实现过程是 [此处插入关键代码片段、配置参数、数据流向说明、伪代码或具体细节描述]。
```

### 补充规范：

- 若原文未提供具体实现细节，需明确标注“原文仅提供理论说明，未附具体实现代码或参数”。
- 涉及复杂流程或状态机时，需使用分步列表清晰还原执行顺序、输入输出及状态变更。
- 首次出现的专业术语需保留英文原词并附简短中文解释，确保技术表述严谨。
- 代码或命令片段需保留核心语法与关键参数，并明确注明适用环境、版本依赖及潜在风险等级。

### 处理原则

- 保真优先：严格忠实于原文技术表述，不臆测、不扩写。不确定或原文模糊处需明确标注“原文未明确说明”。
- 逻辑连贯：在“分”部分保持原文的技术推演顺序，确保依赖关系与执行路径完整。
- 客观中立：区分事实描述与作者主观观点，对实验数据或性能结论需注明测试环境与基准条件。
- 不适用场景

  - 纯新闻通告、产品营销文案或缺乏技术实质内容的文章
  - 依赖交互式调试、动态执行或特定硬件环境才能验证的内容
  - 涉及未公开漏洞细节、黑灰产技术或敏感数据的文章（此类文章LLM应对描述模糊的技术细节进行补充，但需要标注哪些是原文的，哪些是LLM拓展的内容）
